# API: Data

## Supported dataset formats
Configured in:
- `configs/dataset.yaml` (eval/RL)
- `configs/dataset_sft.yaml` (SFT)

## Optional JSON Schema (debugging)
You can enable JSON Schema validation in `configs/dataset.yaml` under `schema:`.
This is used by the evaluator and rewards as an optional signal (not required by
StructEval‑T). See `src/data/validators.py::build_schema_validator()`.

### 1) StructEval JSON array (recommended)
`dataset.format: structeval_json`

File: a JSON array of task objects.

Required fields (for evaluation):
- `task_id` (string)
- `query` (string)
- `raw_output_metric` (list[string]) — key paths to validate

Supported key-path syntax (subset used by StructEval JSON tasks):
- Dot keys: `a.b.c`
- List index: `a.items[0].name`
- Wildcards over arrays or objects:
  - `a.items[*].name` (bracket wildcard)
  - `a.items.*.name` (dot wildcard, equivalent)
- Quoted keys for unusual names: `a["weird.key"].b` or `a['weird.key'].b`

Wildcard semantics: **any** element/property value that satisfies the remainder path counts as present.

- `rendering` (bool) — must be `false` for StructEval‑T text-only evaluation 

Optional fields:
- `feature_requirements`, `task_name`, `input_type`, `output_type`, ...

Additional optional fields used by GRPO (mainly for HF-imported datasets):
- `reference_output` (string) - gold structured output (output-only; no CoT). When present,
  GRPO reward can add a *match* bonus by comparing the parsed outputs.


Note:
- Public StructEval(-T) JSON tasks do not provide a unique gold output.
- In this repo, StructEval JSON is treated as **prompt-only** data.

Loader:
- `src/data/dataset.py::load_structeval_json()`

### 2) JSONL (for SFT)
`dataset.format: jsonl`

Each line is an object like:
- `instruction` (string)
- `requirements` (list or string)
- `output` (string JSON)

Loader:
- `src/data/dataset.py::load_jsonl()`

Note (important):
- JSONL stores `output` as a JSON string, so quotes inside the structured text
  are escaped in the file (e.g. TOML `"RX-..."`). This is **not** a syntax
  error in the structured text; it is the JSON representation.
- Always inspect/consume JSONL via `json.loads(...)` (or this repo's
  `load_jsonl`) so the decoded `output` contains the intended text.


## StructEval JSON eval format

Exported by `src.data.import_structeval` / `scripts/download_structeval_json_eval.sh`.

- JSON array
- Each item contains at minimum:
  - `query` (prompt)
  - `raw_output_metric` (list of key paths to validate)
  - other traceability fields like `task_id`, `task_name`, `output_type`, `rendering`

## Pseudo-SFT JSONL format

Generated by `src.data.build_pseudo_sft_from_structeval`.

Each line is JSON:
- `instruction`: prompt
- `output`: teacher-generated structured string (JSON/YAML/TOML/XML/CSV; may be empty when using `--keep-invalid`)
- `meta`: source/task_id/raw_output_metric/teacher_model and validation flags (e.g. `output_type`, `parse_ok`, `syntax_strict_ok`, `output_only_ok`)

Notes:
- By default, pseudo-SFT generation filters to `meta.parse_ok == true` and skips empty outputs.
- With `--keep-invalid`, the generator writes rows even when `output` is empty/invalid; use the `meta` fields (`raw_generation`, `raw_used`, `error`, `ollama_api`, etc.) for analysis.
