Qwen/Qwen3-4B-Instruct-2507 をベースに、StructEval_TでJSON, YAML, TOML, XML, CSVの推論を評価するためのモデルを作ろうとしています。

仕様書とコードをセットで更新してきているのですが、途中で仕様書の更新が中途半端になっている可能性があります。
コードはgitで管理していて、commitしてdiffを確認しながら進めることにしています。

プロジェクト開始時には学習データは提供されていなかったため、StructEval_TのJSON, YAML, TOML, XML, CSVデータをダウンロードして、Ollamaのgpt-oss: 20bを使ってSFT用の合成データを作成する能力を持っています。structevalのデータは各50件しかないので、SFT用の合成データは4回作ることにしています。
4回というのは、configs/dataset_sft.yamlの中に variants_per_task: 4 というのがあります。ここで1タスクあたりの生成件数を設定しています。
このデータを使ってSFTやGRPOで学習することができるスクリプトがあります。
scripts/sft.sh
scripts/gpro.sh
また、設定ファイルは
configs/dataset_sft.yaml
configs/dataset_grpo.yaml
configs/sft.yaml
configs/grpo.yaml
にあります。

これらのデータセットを、SFTやGRPOの学習に合致するようにデータセットを作成しています。
u-10bei系は
metadataのprompt
metadataのoutput
を学習に使います。u-10bei系のmessagesは会話ログなので、StructEval T向けの学習には使いません。

daichira系は
messagesの
role=userのcontent
role=assistantのcontent
を学習に使います。

学習データのタスクの分布に偏りがあるため、この偏りのまま学習すると、JSON構造の出力に引っ張られる形で、YAMLのインデントミスが発生したり、TOMLのデータ件数が少ないせいで、TOMLへの変換タスクだけ文法ミスが多発しています。
そこで、学習データをタスクごとに用意して、各タスクから均一にデータセットを抽出しています。

SFTやGRPOでの学習は以下にあります。
scripts/run_sft_hf.sh
scripts/run_grpo_hf.sh
このスクリプトは、huggingfaceのデータがなければ、データをダウンロードする処理もおこなっています。

SFTでの学習はforwardの戻り値のpast_key_valuesを返すことでメモリ肥大化につながっているので、use_cache=Falseにしています。
GRPOではキャッシュしたほうがいいようなのでキャッシュしています。

データのダウンロード、SFT合成データ作成、SFT学習、GRPO学習までの方法はREADME2.mdに書いてある通りに実行しています。

文法を正しく理解できるようにするために、学習時はJSON, YAML, TOML, XML, CSVごとに評価方法を用意してあります。
また、出力されたJSON, YAML, TOML, XML, CSVの文法チェックはlintや、出力文字の前後に余計な文字列が入っていないことを確認するアルゴリズムにしています。
学習データに文法ミスがあった場合は学習データから弾くようにしています。
また、u-10bei系とdaichira系のTOMLのデータを求めるタスクの解答がcanonicalに統一されていないせいで、TOMLの推論結果で文法がごちゃまぜになってしまう問題が発生します。
そこで、huggingfaceの学習データで、文法ミスを除去したあとに、tomlデータに関してはcanonicalかどうかを判定して、canonicalでなければcanonicalに変換することを実施しています。

今の問題は、huggingfaceのデータセットは、間違った解答をするタスクが混入していて、品質がわるいことです。
そのため、別途データクレンジングするプロジェクトを立ち上げて、クレンジングした高品質のデータを用意しました。
そこで、configs/sft_hf.yamlやconfigs/grpo_hf.yamlでは、データセットの選び方をわかりやすく設定できるようにしたいと考えています。

online_dataset: 
  use: true/false
offline_dataset:
  use: true/false
このような感じで、オンラインデータセット＝ダウンロードして文法チェックして、タスク分けして、というこれまでの手法をオンラインデータセットとして考えたいです。
今のoffline_datasetの概念は
  use_extra_datasets: false
  extra_datasets:
    - data/my_sft_dataset.jsonl
  extra_split:
    valid_ratio: 0.1
    seed: 42
  extra_filters:
    # null => disable depth filtering
    toml_min_depth: null
となっていますが、これをもっとわかりやすくしたいということです。
offline_dataset:
  enabled: true/false
  datasets:
    - data/my_sft_dataset.jsonl  # 単一ファイル指定
    - filtered_datasets  # ディレクトリ指定。この下にあるすべてのjsonlが対象
  split:
    valid_ratio: 0.1
    seed: 42
  families:
    u10bei:
      targets:
        TOML: all
        JSON: all

        XML: all
        CSV: all
    daichira:

        TOML: 100

たぶんこんな感じになると思います。パラメータ管理構造について、現在のものからわかりやすい構造に変更することを考えてください。
